# Idea Brief / URD — 碑帖释文 chinese-stele-shiwen

> 这是一个纯数据仓库项目，与 [chinese-poetry](https://github.com/chinese-poetry/chinese-poetry) 定位类似，目标是收集中国书法碑帖的**释文**（通行横排整理文本），以 JSON 格式存储，方便开发者引用。

## 目标用户

- **URD-ROLE-001**：前端/小程序开发者——想做一个"每日一碑"App，需要碑帖释文数据
- **URD-ROLE-002**：书法/金石学研究者——需要快速查找某碑的通行释文
- **URD-ROLE-003**：中文 NLP/古文数字人文研究者——以碑帖释文作为语料

## 核心任务

| ID | 说明 |
| --- | --- |
| UR-001 | 提供 JSON 格式的碑帖释文文件，每碑一文件 |
| UR-002 | 按朝代/作者/书体等维度组织目录结构 |
| UR-003 | 为每个释文标注出处（依据哪本书/哪个版本做的释文） |
| UR-004 | 提供统一的 JSON Schema 文档，说明字段含义 |
| UR-005 | 提供 README（含引用方式、贡献指南、数据格式说明） |

## 范围

### in-scope

- 碑帖释文的 **纯文本** 通行整理（竖排转横排后的可读文本）
- 每碑的元数据：碑名、朝代、作者（书丹者）、刊立年份、出处/资料来源、书体
- 标准化的目录结构 + JSON Schema 文档
- 一份可以 `npm install` 或直接 git clone 使用的数据产品

### out-of-scope

- 不做碑帖图片高清扫描图（存链接可以，但本项目不托管图片）
- 不做 OCR 识别、不做智能断句、不做白话翻译
- 不做行列矩阵/原碑字序还原（纯文本版，不做精排）
- 不做 Web 搜索/查询平台（仅是数据仓库）

## 非功能性要求

| 约束 | 说明 |
| --- | --- |
| URD-CON-001 | JSON 文件必须是 **UTF-8** 编码、**无 BOM** |
| URD-CON-002 | 每碑独立 JSON 文件，方便按名引用；不做一个大 JSON 合集 |
| URD-CON-003 | 数据可离线使用，零依赖（纯文本文件） |
| URD-CON-004 | Schema 向后兼容——只在末尾加字段，不删不改旧字段 |
| URD-CON-005 | 整个仓库（含元数据 CSV 索引）控制在合理大小（目标初版 < 1000 碑） |

## 成功标准

| ID | 标准 |
| --- | --- |
| URD-AC-001 | `data/stele/` 目录下有 ≥ 50 种代表性碑帖的真实释文 JSON 文件（初版目标） |
| URD-AC-002 | 任何文件都可通过 `jsonschema` 校验 |
| URD-AC-003 | README 读完后，外部贡献者能独立新增一个碑帖并提交 PR |
| URD-AC-004 | 仓库能 `git clone` 后直接被 `import json` / `fetch` 引用，无需 build 步骤 |

## 假设

- URD-ASM-001：释文资料来源于公开出版物（如《金石萃编》《北京图书馆藏中国历代石刻拓本汇编》等），整理者注明出处即可
- URD-ASM-002：初版释文以通行简体横排为主，不要求繁体/异体对照
- URD-ASM-003：零运行时依赖（纯 JSON 文件），数据处理脚本可选 Python 但不强制

## 开放问题

- URD-Q-001：按朝代分目录 vs 按书体分目录？——默认：按朝代（与 chinese-poetry 风格一致）
- URD-Q-002：一个碑帖的不同释文版本是否要同时保留？——初版只收一个通行版
- URD-Q-003：数据是否需要配套全文检索索引？——初版不做，留作 PARKING_LOT
